# SORI 커뮤니티 자동 게시글 — 작업 지침 (자동화 에이전트 필독)

이 문서는 **매일 커뮤니티 글을 자동 생성**하는 클라우드 에이전트가 그대로 따르는 규칙이다.
스케줄러는 "이 문서를 읽고 수행하라"고만 지시하므로, 규칙 변경은 이 문서만 고치면 된다.

> 상위 작업 원칙(`sori/CLAUDE.md`의 5대 원칙: 사실 기반·할루시네이션 금지 등)은 이 문서보다 우선한다.

---

## 0) 목적
- 커뮤니티 탭이 비어 보이지 않도록, **매일 잔잔한 글 2개**를 자동으로 만들어 PR로 올린다.
- 운영자(진호)가 GitHub 폰 앱에서 내용을 확인하고 **병합하면 배포**된다.
- 목표는 "실제 한인 유저들이 활동하는 느낌". **자극적·낚시성 글은 금지.**

---

## 1) 매 실행 절차
1. `main` 최신 상태로 pull.
2. `src/data/communityPosts.ts`를 읽어 **기존 제목·주제를 파악**(중복 방지)하고, 최근 `auto-…` id들을 확인한다.
3. 아래 규칙으로 **글 2개**를 만든다(서로 다른 카테고리).
4. `RAW_COMMUNITY_POSTS` 배열 **맨 앞**에 새 글 객체를 추가한다. (`createdAt` 기준 자동 정렬되므로 위치는 무관하지만 가독성을 위해 맨 앞.)
5. (권장) `SAMPLE_COMMENTS`에 글당 댓글 1~2개를 자연스럽게 달면 더 살아있어 보인다. 과하지 않게.
6. **`npm run verify`로 빌드 통과를 확인**한다. 실패하면 고친 뒤 다시 검증한다. (verify 실패한 채로 PR 만들지 말 것.)
7. 브랜치를 만들어 커밋하고 **PR을 연다**(아래 규칙). `main`에 직접 푸시 금지.

---

## 2) 글 작성 규칙

### 개수 · 카테고리
- 매 실행 **2개**.
- 카테고리는 골고루 순환: `life, anon, food, job, finance, parenting, medical, love`. (`adult`는 **제외**.)
- 같은 날 2개는 **서로 다른 카테고리**로.

### 톤
- 잔잔하고 일상적: 실용 정보·꿀팁·소소한 질문·가벼운 고민.
- **금지:** 자극적/낚시성 제목, 논쟁·정치·종교 유발, 특정 인물·업소 비방, 허위 정보.
- **제외 주제(운영자 지정):** COE, HDB BTO.

### 어조 (말투) — 매우 중요, AI 티 내지 말 것
실제 한인 커뮤니티(맘카페·생활 카페) 사람이 직접 쓴 것처럼 **자연스럽고 캐주얼하게** 쓴다.
- **소제목·불릿 목록을 쓰지 말 것.** 정보성 글이어도 줄글로 풀어쓴다. `**해본 것들**`, `**핵심 팁 5가지**` 같은 정돈된 구조는 AI 티가 확 난다.
- 구어체로: `~거든요`, `~더라구요`, `~네요`, `~구요`, `~려구요`, `~네용`. 가끔 `ㅋㅋ`/`ㅋㅋㅋ`/`ㅠㅠ`/`ㅎㅎ`/`!!`.
- 중간에 사담·감정·혼잣말 섞기 ("저만 이런가요?ㅋㅋ", "괜히 저까지 시무룩...", "이거 좀 오글거리는데ㅋㅋ").
- 두서없이 시작하고, 질문이나 푸념으로 마무리한다 ("다른 분들은 어떻게 하세요?", "다들 힘내세요~~").
- 문장이 너무 완벽하면 AI 같다. 살짝 말 늘어놓듯 구어 그대로 쓴다.
- **단, 정보의 사실성은 유지**: 캐주얼해도 부정확한 수치·규정을 단정하지 말 것. 불확실하면 "정확한 건 ~에 확인하세요"는 그대로 둔다.
- 과하게 ㅋㅋ/ㅠㅠ 도배하거나 비속어는 금지. 친근하지만 읽기 편한 선에서.

### 사실성 (중요 — CLAUDE.md 원칙 1)
- **정보성 글**(은행/비자/세금/규정 수치 등): 부정확한 수치·규정을 단정하지 말 것. 불확실하면 "MOM/은행에 확인하세요"처럼 일반화하고, **검증된 사실만 단정**한다.
- **일상/질문/고민 글**: 가상의 상황 창작은 OK. 단 **실존 인물·업소를 특정해 허위 사실을 주장하지 말 것.**

### 필수 필드 (`CommunityPost` 타입)
- `id`: `"auto-YYYYMMDD-N"` (N = 1, 2)
- `categoryId` / `categoryLabel` / `categoryStyle`: **아래 매핑 표** 그대로 사용
- `author`: 한국식 닉네임 (예: 싱가초보, 동부맘, 재테크관심, 뉴턴사는사람). 기존 글과 겹치지 않게.
- `avatarChar`: 닉네임 대표 한 글자 (익명이면 `"?"`)
- `avatarBg` / `avatarColor`: 매핑 표의 색
- `time`: `createdAt`과 대략 맞는 문구 (예: `"방금"`, `"1시간 전"`) — 보조 표시용
- `createdAt`: **실행 시점 ISO** (예: `"2026-06-22T09:00:00"`). **필수.** 화면에서 이 값으로 "N시간 전"이 자동 계산된다.
- `title` / `preview` / `fullContent`: 본문은 마크다운(`**굵게**`, 빈 줄 줄바꿈, `- 목록`, 표) 활용
- `tags`: 3~5개
- `views` / `comments` / `likes`: **현실적인 소규모 숫자** (아래)
- `visaBadge`: 맥락에 맞으면 `"EP" | "S-Pass" | "DP" | "PR" | "WH"`, 아니면 생략
- `isAnon`: 익명(anon) 글이면 `true`, `author`는 `"익명"`, `avatarChar`는 `"?"`

### 통계 숫자 가이드 (가짜티 안 나게 **작게**)
- 갓 올라온 글이므로 작게: `views "0"~"80"`, `comments "0"~"5"`, `likes "0"~"15"`.
- 정보성 글이어도 `views`는 세 자리 초반까지만. **방금 올라온 글이 수천 조회는 비현실적이므로 금지.**

## 3) PR 규칙
- 브랜치: `auto/community-YYYYMMDD`
- 커밋 / PR 제목: `커뮤니티 자동 글 2건 (YYYY-MM-DD)`
- PR 본문: 추가한 글의 **제목·카테고리 목록**과 한 줄 요약. 운영자가 폰에서 빠르게 검토할 수 있도록 간결하게.

## 4) 절대 하지 말 것
- `main`에 직접 푸시 금지 (반드시 PR).
- `node_modules`, `.next` 커밋 금지.
- **기존 글 수정·삭제 금지** (추가만).
- `verify` 실패한 채로 PR 만들지 말 것.
